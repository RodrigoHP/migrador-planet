# Story 4.1: Correção de Rotas e Fluxo Upload → Extraction

## Status

Ready

## Story

**As a** operador da ferramenta,
**I want** que o upload de arquivos e o início do processamento funcionem corretamente ao clicar "Analisar",
**so that** o fluxo completo de upload → pipeline → progresso SSE → navegação para /campos seja funcional end-to-end.

## Acceptance Criteria

1. `POST /api/upload/pdf` retorna `{ "jobId": "<uuid-v4>" }` — o jobId é gerado no primeiro upload e reutilizado pelas demais chamadas da mesma sessão
2. `POST /api/upload/xsd` aceita `jobId` como campo `Form` e salva `schema.xsd` no diretório correto do job
3. `POST /api/upload/data` aceita `jobId` como campo `Form` e salva `data.json` ou `data.xml` no diretório correto do job
4. `POST /api/jobs` com `{ "jobId": "<uuid>" }` inicia o pipeline em background e retorna `{ "jobId": "<uuid>" }` imediatamente
5. Frontend (`UploadPage.vue`) chama todos os uploads com prefixo `/api/`: `/api/upload/pdf`, `/api/upload/xsd`, `/api/upload/data`
6. Frontend extrai `jobId` do response do upload PDF e o envia no body de `POST /api/jobs` (elimina a chamada a `POST /extract`)
7. `useSSE.ts` conecta em `/api/progress/${jobId}` (com prefixo `/api/`)
8. SSE stream emite `step`, `done` e `error` — frontend atualiza barra de progresso corretamente
9. Ao receber evento `done`, frontend popula `session.extraction` com o resultado e navega para `/campos`
10. Nenhuma chamada de rede retorna 404 por problema de roteamento de prefixo

## Dev Notes

### Causa Raiz dos Gaps G1–G5

**G1–G4 (prefixo /api ausente):**
O Vite proxy intercepta apenas rotas com prefixo `/api`:
```ts
// vite.config.ts
proxy: {
  '/api': { target: 'http://localhost:8000', changeOrigin: true }
}
```
O frontend chamava `/upload/pdf` (sem `/api`) — o Vite não proxy-eava, a chamada ia direto para o frontend server e recebia 404. Solução: adicionar `/api/` em todas as chamadas.

**G5 (rota /extract vs /api/jobs):**
O frontend chamava `POST /extract` esperando receber `{ jobId }` de volta. O backend nunca teve esse endpoint — tem `POST /api/jobs` que recebe `{ jobId }` no body. A solução correta é:
1. Upload PDF gera o `jobId` e o retorna
2. Frontend usa esse `jobId` para todas as chamadas subsequentes
3. Frontend chama `POST /api/jobs` com `{ jobId }` para iniciar o pipeline

@router.post("/upload/pdf")
async def upload_pdf(file: UploadFile = File(...)):
    job_id = str(uuid.uuid4())
    job_dir = TMP_BASE / job_id
    job_dir.mkdir(parents=True, exist_ok=True)
    content = await file.read()
    (job_dir / "input.pdf").write_bytes(content)
    return {"jobId": job_id}

# upload.py — POST /api/upload/xsd MODIFICADO
@router.post("/upload/xsd")
async def upload_xsd(
    file: UploadFile = File(...),
    jobId: str = Form(...),
):
    job_id = _validate_job_id(jobId)
    job_dir = get_job_dir(job_id)
    content = await file.read()
    (job_dir / "schema.xsd").write_bytes(content)
    return {"ok": True}

async function requestExtraction(jobId: string): Promise<string> {
  const response = await fetch(`${API_BASE}/api/jobs`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ jobId }),
  })
